# 代码审查报告（2026-07）

> 对全库五个领域（agent 层 / adapter·client·stream / conversation / external 栈 /
> facade·model·公共 API）的一次完整设计与实现审查。每条发现给出文件路径与行号。
> 严重程度分级：🔴 高（正确性/安全 bug，建议尽快修）、🟡 中（设计承诺与实现脱节、
> 健壮性缺口）、🟢 低（打磨项）。

## 总体评价

架构底子是好的：

- **sans-io 纪律执行严格**——`agent/machine/` 与 `agent/external/machine.rs` 全文无
  tokio/IO/系统时钟依赖，effect 以 `Requirement` 外化，IO 全部在 handler/driver 侧。
- **Conversation 不变量分层清晰**——committed log + pending + projection 三层写入设施
  被 revert/fork、compaction、cancel 共用，与 DESIGN.md 的设计一致；cancel 原子性、
  Boundary ABA 防护（owner+version+anchor+ceiling 四重校验）经核查无问题。
- **SSE 解析完备**——截断/EOF/乱序/重复事件均有显式错误路径，无裸 unwrap，JSON 错误
  带行列号。
- **多数凭证脱敏到位**——HTTP 认证头全部 `set_sensitive(true)`，external config 的
  Debug redact env 值，probe 错误不回显 CLI 输出，spawn 全部走 argv 无 shell。

问题集中在三类：**安全/健壮性硬 bug**、**设计承诺与实现脱节**、**大面积复制代码
导致同一缺陷需要修 N 遍**。

## 🔴 高严重度

### 安全

#### H-SEC-1. API key 可经派生 `Debug` 明文泄露 ✅ 已修复（M1-1）

`EndpointConfig` 与 `AuthScheme` 均派生 `Debug`（`src/client/config.rs:9,36`），两个
adapter 同样 derive `Debug` 并内嵌 `endpoint`（`src/adapter/anthropic/mod.rs:22-26`、
`src/adapter/openai_resp/mod.rs:25-29`）。`tracing::debug!(?adapter)`、`dbg!`、anyhow
context 都会原样打印 `auth: Bearer("sk-...")`。`config.rs:33-35` 的文档只警告了 serde
路径，更常见的 Debug 路径没有防护。

对照：HTTP 请求侧做对了（认证头 `set_sensitive(true)`，`anthropic/request.rs:292`）；
facade 的 `ProviderConfig` 也做了脱敏——策略不统一。修法：手写脱敏 Debug（约 30 行）。

#### H-SEC-2. HTTP 默认无超时 + 错误 body 无界读取 → 可永久挂起 ✅ 已修复（M1-2）

两个 adapter 的 `new()` 使用裸 `reqwest::Client::new()`（`anthropic/mod.rs:31`、
`openai_resp/mod.rs:34`），无任何超时；`ClientError::Timeout` 变体在默认配置下永远不
会产生。更严重的是错误路径：HTTP 非 2xx 时无条件读全量 body
（`anthropic/stream/mod.rs:48`、`openai_resp/stream/mod.rs:47`、两个 `response.rs:67`）：

```rust
let body = response.bytes().await.map_err(map_transport_error)?;
```

对端返回错误状态码但保持连接不关闭时，此 future 永不就绪。建议：默认 client 带合理
总超时 + connect 超时；错误 body 加大小上限（如 1 MiB）与独立超时。

#### H-SEC-3. `Usage::merge` 溢出即 panic，可被 wire 数据触发 ✅ 已修复（M1-3）

`src/model/usage.rs:138-141`（审查时点行号）：

```rust
fn checked_add(left: u32, right: u32, field: &str) -> u32 {
    left.checked_add(right)
        .unwrap_or_else(|| panic!("usage {field} token count overflowed u32"))
}
```

`Accumulator::push` 对每条 `StreamEvent::Usage` 调用 `merge`
（`stream/accumulator/mod.rs:146`）；facade 的 `UsageSummary` 聚合（`facade/run.rs:249-269`）
同样经过它。异常/恶意端点上报伪造大计数即可让宿主进程 panic。库代码不应因 wire 数据
panic，应改 `saturating_add` 或返回错误。

#### H-SEC-4. `ClientError::Network` 可能携带含凭据的完整 URL ✅ 已修复（M1-4）

4 处 `map_transport_error` 副本（`anthropic/stream/mod.rs:89`、`anthropic/response.rs:173`、
`openai_resp/stream/mod.rs:88`、`openai_resp/response.rs:163`）均
`ClientError::Network(error.to_string())`。reqwest 错误 Display 形如
`error sending request for url (https://host/path?api-key=...)`——含完整 URL。若部署把
凭据放进 `query_params`（Gemini 风格 `?key=`），key 会进入错误消息与日志。建议拼 URL
后或构造错误时对 query 脱敏，或在 `EndpointConfig` 文档明确禁止 query 放 secret。

### 外部 CLI 子进程

#### H-EXT-1. 流读取超时复用 30 秒 launch 超时，长静默 turn 被误杀 ✅ 已修复（M1-5）

三个 adapter 都把 `config.timeout()`（默认 **30 秒**，本为 probe/launch 设计）同时用作
每行 stdout 读取超时（`claude_code/adapter.rs:168-169`、`codex/adapter.rs:222-223`、
`opencode/adapter.rs:236-237`）：

```rust
read_timeout: config.timeout(),
shutdown_grace: config.timeout(),
```

CLI 执行 5 分钟无输出的命令（跑测试套件、`cargo build`）→ 30 秒无新帧 →
`SessionLost` → session 被杀、worktree 标记残余副作用。这是**语义混淆**而非配置问题：
`timeout()` 的文档只说它是 probe/launch timeout。真机必踩。应为 streaming inactivity
单设一个更长（或无界 + 心跳）的超时。

#### H-EXT-2. kill 无 process group——孙进程泄漏 ✅ 已修复（M2-1）

全部进程管理只有 `start_kill()` / `kill_on_drop(true)`；
`grep "process_group|setsid|killpg" src/` 零命中。SIGKILL 只发给 CLI 本身，CLI 经 Bash
工具拉起的子进程（构建、测试、dev server）kill 后成为孤儿继续运行，可能继续写已被
`git worktree remove --force` 删除的目录、持有端口/文件锁。与 §6.4 "force-close 后
worktree 不可复用" 的模型自相矛盾。修法：spawn 时 `process_group(0)`，kill 时杀进程组。

#### H-EXT-3. `close()` 对任何退出码一律分类为 `Graceful` ✅ 已修复（M1-6）

`claude_code/adapter.rs:198-199`（codex/opencode/acp 同构）：

```rust
match timeout(self.shutdown_grace, self.child.wait()).await {
    Ok(Ok(_status)) => ExternalSessionShutdown::Graceful,   // ← 不看 exit code
```

CLI 以 exit code 1 崩溃也算 Graceful；下游 `worktree.rs:470` 据此判定 ephemeral
worktree 是否"干净可删/可复用"——崩溃 session 会把写了一半的 worktree 判为干净。
应 `status.success()` 才 Graceful，否则 Failed。

### 状态一致性

#### H-STATE-1. reverted head 上做 compaction 永久破坏投影覆盖 ✅ 已修复（M3-1）

`apply_compaction` 只校验 `plan.head_turn_count == active_len`
（`src/conversation/projection/compaction.rs:251-258`），**不校验 head 是否等于 lineage
上限**。可复现路径（5 个 turn，已压缩 0..5）：

1. `revert_to(boundary 3)` → head=3，投影仍覆盖 0..5；
2. 在 head=3 上 `apply_compaction` → 新投影只覆盖 0..3，旧覆盖被丢弃；
3. `valid_boundaries()` 仍枚举 0..=5，redo `revert_to(boundary 5)` 成功，而 `revert_to`
   不触碰投影；
4. `effective_view()` 只遍历 spans——**turn 3..5 的消息被静默省略**，发给 LLM 的上下文
   永久缺一段。

且无法自愈：再 compact 会因 `IncompleteProjection` 失败；后续 commit 走
`extend_after_commit`（无覆盖校验）空洞永存；snapshot → restore 会因 `SpanGap` 被拒绝。
建议：compaction 拒绝 `active_len != lineage_len`，或 redo/commit 时把空洞物化为 raw
span。需补回归测试（revert → compact → redo → 校验 effective_view 完整）。

#### H-STATE-2. rows 持久化丢失 `MessageMeta` ✅ 已修复（M3-2）

`MessageRecord` 只有 payload 没有 meta（`src/conversation/persistence/rows.rs:123-133`）：

```rust
pub struct MessageRecord {
    pub message_id: MessageId,
    pub turn_id: TurnId,
    pub message_sequence: u64,
    pub payload: Message,          // ← 没有 meta 字段
}
```

`inject_user_message` 经 `new_with_meta` 写入的 envelope 元数据（pivot 注入消息的
source）在 `to_rows() → into_snapshot()` round-trip 后丢失，与模块文档"describe the
same committed consistency point as ConversationSnapshot"矛盾。现有 e2e 断言因夹具未用
`inject_user_message` 从未覆盖。修复：`MessageRecord` 增加 `meta: Option<MessageMeta>`。

#### H-STATE-3. facade `run_full` 无取消防护：一次 timeout 永久毒化 Agent ✅ 已修复（M5-1）

非流式路径直接 `drain(&mut self.machine, ...).await?`（`src/facade/agent.rs:354`），无
Drop 守卫。流式路径专门设计了 `AgentRunStream::abandon`
（`facade/agent/stream.rs:496-521`）回收滞留 turn，`run_full`/`run` 没有等价物。调用方
用 `tokio::time::timeout` / `select!` 包裹 `agent.run(..)` 超时后，machine cursor 停在
携带 outstanding requirement 的中间态，下一次 run 以 protocol error 失败——**Agent 被
一次超时永久毒化**。facade 文档声称 "A failed turn discards its uncommitted work … the
`Agent` stays usable"（agent.rs:268-269），这只对 `Err` 返回成立，对 future 被 drop 不
成立。

#### H-STATE-4. pivot 重发同一 requirement id 与 trace 去重冲突 ✅ 已修复（M4-3）

机器 pivot 路径用同一个 requirement id 重发 LLM 请求
（`machine/default/mod.rs:614-623`，注释明确 "re-emitted under the *same* requirement
id"），而 drain 把 requirement id 直接当 trace node id（`drive.rs:445-494`），
`TraceHandle::record_node` 对重复 id 返回 `DuplicateNodeId`（`context/trace.rs:379-382`）
并经 `?` 中止整个 drain（`drive.rs:425-430`）。任何支持 pivot 且记录 trace 的 driver，
第二次 fulfill 同一个 LLM requirement 时必然 turn 中途失败。

更广义的问题：**纯观测侧的 trace 失败会杀死实际驱动**——trace 应该是 best-effort。

#### H-STATE-5. 停在 `AwaitingReconfig` 期间新 reconfigure 请求被静默丢弃 ✅ 已修复（M4-2）

`DefaultAgentMachine::reconfigure`（`machine/default/mod.rs:314-319`）无 cursor 守卫，
任何时候可入队；resume 路径应用的是 park 时预先 plan 好的 application，而
`apply_reconfig_application`（`state.rs:200-208`）**无条件 `queued_reconfigs.clear()`**。
场景：队列 [R1] → park → host 调 `reconfigure(R2)`（校验通过入队 [R1,R2]）→ resume →
只应用 A1 并清空队列——R2 已校验通过却既不生效也不报错，静默丢失。

#### H-STATE-6. `blackboard_read` 工具丢弃消息正文；mailbox 无读工具 ✅ 已修复（M4-1）

`src/agent/collab/tools.rs:587-598`：

```rust
let messages = self.blackboard.read_from(&channel, from);
Ok(Ok(format!(
    "read {} message(s) from `{channel}` offset {from}",
    messages.len()
)))
```

消息取出后只数个数就丢弃——黑板对模型**只写不读**。Mailbox 同样只有 `send_message`
没有收件箱 bridge 工具。协作协议（agent-layer.md §6.4）在模型侧断裂。属一行级 bug。

## 🟡 中严重度

### 承诺与实现脱节

#### M-PROM-1. 预算在默认 agent 路径完全未执行

`agent-layer.md` §1.4 要求"每步检查预算"，但 `charge_step`/`charge_usage`/
`charge_tokens`/`charge_cost_micros` 在默认路径（drain、drive_turn、facade
`drive_streamed`）零调用，只有 external/ 和测试在用；facade 恒构造
`BudgetLimits::unbounded()`（`facade/agent/stream.rs:203-207`）。LLM `Response` 的
usage 被 fold 后从未计费。连带 `CancelRecoveryReason::BudgetExceeded`、
`LoopDoneReason::BudgetExhausted`（`state/cursor.rs:615,654`）成为死变体。要么补实现，
要么改文档承认现状。

#### M-PROM-2. facade 无法触达下层的 cancel / budget / pivot（cancel/pivot ✅ 已修复（M5-4）；budget 待 M6-2）

每次 run 都 `RunContext::new_root(..)` 新建私有 `CancellationToken`
（`facade/agent.rs:298-303` 等），facade 无 `cancel()` 方法、不保留 token、无 budget
旋钮、无 pivot 入口。`ToolContext.cancel` 因此是永远不会被取消的令牌。
`docs/facade-api.md` §13 草拟了 cancel API 形态但实现缺失，且未在文档标注"尚未暴露"。

#### M-PROM-3. pivot 在所有随库 driver 中不可达

`drain`/`drive_turn` 只接收一个 `AgentInput` 一路跑到 terminal
（`drive.rs:369-438`），无中途喂输入通道；facade 全文无 `Pivot` 使用。`inject_pivot`
对库内所有 driver 是死路径，自定义 driver 一旦触达又会撞上 H-STATE-4。

#### M-PROM-4. 审批拒绝的文档承诺与 typed-tool 实际行为相反 ✅ 已修复（M5-2）

文档承诺 deny 时 "the run surfaces `FacadeError::ApprovalDenied`"
（`facade/approval.rs:190-193`、`facade/error.rs:71-80`）。实际普通 typed tool 被拒绝时
machine 合成 `ToolStatus::Denied` 的 tool result 回灌模型、循环继续、run 正常 `Ok`
（`machine/default/tools.rs:535-542`；测试 `auto_deny_skips_tool_execution` 明确断言
成功）。`ApprovalDenied` 只在 external delegate 路径真正抛出。按文档 `match` 该错误的
调用方永远不触发。两处公共文档需修正（或改行为）。

#### M-PROM-5. `ExternalSessionPolicy` 被所有 adapter 忽略；`WorktreeManager` 未接入生产路径 ✅ 已修复（M2-7）

machine 每个请求携带 `policy: *spec.session_policy()`（`machine.rs:584-585`），但
`grep request.policy src/agent/external` 只命中测试——请求级 `permission_mode`/
`max_turns` 静默失效，同一 adapter 无法服务两个不同权限策略的 agent。另外
`GitWorktreeManager` 在 `src/` 生产路径无人调用——worktree 隔离只存在于
`examples/support/managed.rs:367-394`。**AGENTS.md 宣称的 "every managed child runs in
a throwaway worktree" 是 example 的行为，不是库的保证。**

修复（M2-7）：`ExternalSessionRegistry` 持有 `Arc<dyn WorktreeManager>`（默认
`GitWorktreeManager`），start/resume 前按 `request.policy.isolation` 调 `prepare` 并把
prepared 路径写入新增的 `ExternalSessionRequest::session_dir` 交给 adapter（优先级高于构造期
`config.working_dir`，OpenCode 贯通到 `--dir`）；registry 记录每个 live session 的
`PreparedWorktree`，`cleanup`/`cleanup_agent` 按 session disposition 调
`WorktreeManager::cleanup`。`permission_mode` 由四个 adapter 在 start/resume 以请求级覆盖
构造期 config；`max_turns` 由 machine 统一强制为 runtime round-trip 上限（`LimitExceeded`）。
详见 `docs/managed-external-agent.md` §16。

#### M-PROM-6. facade 死配置：`provider_extras` 不可达 ✅ 已修复（M5-5）

`ModelConfig::provider_extras(...)` 存在且 `apply_to_request` 会传递
（`facade/config.rs:400-405`），但所有 builder（Chat/Agent/AgentWorker/AgentRestore）
都不暴露它，`build_request` 恒为 `provider_extras: None`（`chat.rs:235`）。facade 用户
实际无法发送 `top_k`、reasoning effort 等关键请求参数。

### 错误语义与取消

#### M-ERR-1. 所有 step 错误都是破坏性的：driver 协议失误 = 整个 turn 被销毁 ✅ 已修复（M4-4）

`machine/default/mod.rs:1048-1058`：任何 `StepError` → `fail` →
`cancel_pending(DiscardTurn)` + Error cursor。以下**可恢复的调用方错误**都会丢掉整个
pending turn（含已完成 tool 结果）：driver 喂 stale resume id、在不合法边界喂 pivot
（"用户在第一次 LLM 调用途中又发一条消息"这种正常 UX）、turn 进行中喂第二个
UserMessage。机器没有"拒绝输入、保持状态"的软出口，协议违规与内部故障混为一谈。

连带问题：

- `NestedMachine::route_by_id` 的 fallback（`nested.rs:266-272`）把未知 id 的
  Resume/Abandon 转发给根机"分类错误"——一个迟到的 cancel/resume 会销毁根机正在
  进行的 turn。
- `fail()` 自身的错误被吞（mod.rs:1021-1030）：转移表没有 `(Done|Error) → Error`
  边，机器已停在 Done/Error 时 `fail` 的转移静默失败，错误消息丢失、cursor 停留
  旧状态、outcome 看起来正常。
- 步数上限走 `LoopCursor::Error` 而非为此准备的 `LoopDoneReason::StepLimitReached`
  （`machine/default/tools.rs:593-604` vs `cursor.rs:650`）——正常策略性终止被归类为
  运行错误并丢弃整轮。

#### M-ERR-2. 取消延迟无界；`TurnDone` 契约自相矛盾 ✅ 已修复（M4-4 第 4 条 + M4-5 其余）

- `drain` 只在每批之间检查 `ctx.is_cancelled()`（`drive.rs:405`）；飞行中的 LLM/工具
  调用不可中断，参考实现的 handler 全部忽略 ctx（`reference.rs:91,153` 的 `_ctx`），
  无超时。取消信号到达后 turn 还会再推进一整批。
- 取消时只对批次第一个 requirement 记录 `NeverResumed`，其余静默丢弃、无 trace 记录
  （`drive.rs:405-412`），违反"每个 requirement 恰好以一种方式 settle 并记录"。
- 取消后返回的 `TurnDone` cursor 是 `Idle`，而文档声称是 terminal `Done|Error`；
  调用方无法区分取消与自然结束。
- during-turn reconfig 的 abandon 用 `DiscardTurn` 丢弃整轮（mod.rs:978-986），而
  tool abandon 精心用 `ResumeTurn` 保全工作（tools.rs:701-716）——同样的取消两种
  待遇。

#### M-ERR-3. 重配置的两处 resolver 不一致；默认 resolver 使校验形同虚设 ✅ 已修复（M4-6）

机器 queue 时用 `self.tool_registry_resolver` 校验（mod.rs:327-346），driver apply 时
用 scope 的 resolver（`reference.rs:183-195`）——两个不同对象可配置成不同实现：queue
通过、apply 失败 → 已完成 turn 被销毁，且 reconfig 留在队列里下个 turn 边界再次失败，
形成重复失败循环。默认 `DeclaredOnlyToolRegistryResolver`（`tool.rs:143-152`）对任何
tool set 都"解析成功"但 `execute` 恒 `UnknownTool`——默认接线下 reconfig 全链路"成功"
然后每个工具调用开始失败，footgun。

#### M-ERR-4. HTTP 错误分类顺序导致误分类

`src/client/error.rs:90-99`：对任意状态码先做 body 子串匹配，再判 401/403。403 body
含 "content policy"（如 Azure 权限拒绝文案）会误报为 `ContentFiltered` 而非 `Auth`；
500 body 回声文本含 "too many tokens" 会误报 `ContextLengthExceeded`。重试/降级策略
依赖这些类别，误分类导致错误决策。marker 匹配应限定 4xx 且排除 401/403。

#### M-ERR-5. facade 用错误消息子串匹配做错误分类 ✅ 已修复（M5-3）

`src/facade/agent.rs:1546-1552`：`message.contains("loop step limit")` →
`LoopLimitExceeded`。依赖 `machine/default/tools.rs:601` 的字面量措辞，下层一改就静默
退化为 `Agent(Other)`，且结构化信息（哪个上限、实际值）全丢。应让 `LoopCursor::Error`
携带结构化 kind。

### external 栈

#### M-EXT-1. 跨进程 resume 后事件 seq 从 0 重启，恢复后的事件被静默吞掉 ✅ 已修复（M2-2）

design §5.5 要求 "seq spans the whole session"，但 resume 路径构造全新 decoder
（`claude_code/adapter.rs:645-651`，next_seq=0），而 machine 去重用持久化的旧高水位
（`machine.rs:732-741`：`observed.seq > consumed`）。原 session 消费到 seq=50 → 重启
→ resume 后第一个 decision point 的所有 observation（seq < 50）全部被丢弃，通知流
出现静默缺口。修复：resume 时用 `ExternalSessionRef.last_event_seq` 播种 decoder。

#### M-EXT-2. 子进程全量继承父进程环境，与 ACP 的 `env_clear` 策略不一致

三个 CLI adapter 只 `command.env(k, v)` 覆盖（`claude_code/adapter.rs:152-154` 等），
tokio Command 默认继承全部父 env；ACP 则 `env_clear()` + 可选 inherit
（`acp/connection.rs:215-218`）。宿主的 `DEEPSEEK_API_KEY`、`AWS_*` 等全部进入 CLI
子进程及其拉起的 MCP server 和任意 shell 命令。姿态不一致且 CLI 侧更松。

#### M-EXT-3. decoder 把 runtime 原文塞进错误消息，与模块文档自我承诺矛盾 ✅ 已修复（M2-3）

三个 decoder 文档都声称 "no prompt text, tool input, or credential is ever folded into
an error message"（`claude_code/decoder.rs:48-49`），但实现把
`frame.get("result").and_then(Value::as_str)` 原文放进 `ExternalAgentError::Runtime`
（decoder.rs:517-521；codex/opencode 同构）。该文本受模型输出影响，可包含模型刚 cat
出的 `.env` 内容，经 `Failed` → `error.to_string()` 进入 cursor 与日志。修文档或截断
脱敏二选一。

#### M-EXT-4. Codex/OpenCode 把 prompt 作为位置参数传递 ✅ 已修复（M2-4）

`codex/adapter.rs:121-124`、`opencode/adapter.rs:127-143`：`args.push(prompt.clone())`，
前面无 `--`。问题：(1) prompt（可能含敏感代码）对本机任何 `ps` 可见；(2) 以 `-` 开头
的用户消息会被 clap 当 flag 解析，轻则 launch 失败重则改变 CLI 行为。Claude Code 走
stdin frame 无此问题。

#### M-EXT-5. Codex/OpenCode 决策点到达后不 reap 子进程；关闭 disposition 被丢弃 ✅ 已修复（M2-5）

decoder 在 `turn.completed` 行就返回 decision（`codex/adapter.rs:559-561`），不再读到
EOF；`let _ = old.close().await;`（adapter.rs:462-464）丢弃 disposition——子进程可能
僵尸或阻塞在满管道，close 超时导致的 `ForcedKill` 被吞掉，不进 trace 也不影响
worktree 判定，与 §6.4 审计承诺不符。

#### M-EXT-6. `begin()` prelude 循环无取消检查、无总时限 ✅ 已修复（M2-5）

`claude_code/adapter.rs:298-311`（codex/opencode 同）：per-line timeout 每行重置，不断
吐非 init 帧的 CLI 会让 `start()` 永远循环；advance 循环有 `ctx.is_cancelled()` 检查，
prelude 没有。

#### M-EXT-7. worktree cleanup 把 worktree 路径当 repo 参数传给 `git -C` ✅ 已修复（M2-6）

`worktree.rs:477-485`：`PreparedWorktree` 不保存 base repo 路径，cleanup 只好把
worktree 自己当 `-C` 目录。常规 linked worktree 下碰巧能工作，但目录部分损坏/被移动
时直接失败。应记录 base repo。

### conversation 层

#### M-CONV-1. restore 的派生索引"交叉校验"是空操作 ✅ 已修复（M3-3）

`persistence/snapshot.rs:566-575`：`ToolCallIndex::rebuild(turns, None)` 同一纯函数调
两次再比较，`RestoreError::DerivedIndexMismatch` 不可达——给人"restore 校验了派生
索引"的假象。删除或改为对比增量路径 vs 全量重建。

#### M-CONV-2. 长链递归：restore 校验与 drop 均可能栈溢出 ✅ 已修复（M3-4）

三处递归深度 = 父指针链长（turn 数）：`snapshot.rs:415-440` `visit_parent`（对不可信
快照做环检测）、`history.rs:359-378` `build_restored_node`、`history.rs:352-356` cons
链表的递归 drop（长会话析构 `History` 的经典栈溢出风险）。建议改迭代 + drop 手工逐层
`Arc::try_unwrap` 摘链。

#### M-CONV-3. "insert-only" 行模型无法表示会话演进 ✅ 已修复（M3-5）

`rows.rs:1077-1092`：同一 conversation 第二次导出必然 `InsertConflict`——任何
commit/revert 都改变 `head_turn_count`/`structural_version`；revert 后 lineage 关联行
同序号引用不同 turn；compaction 重写 span 行。该 API 实际只支持"每个 conversation
恰好导出一次"，此限制未在任何文档声明，与"DB-neutral 增量导出"定位不自洽。

#### M-CONV-4. 每个 pending 操作对全量历史线性/平方扫描

`history.rs:211-217` `contains_message_id` 每次物化全量引用扫全部消息，被
begin_turn/finish_assistant/append_tool_*/inject_user_message 在每次 pending 转换时
调用；`pending/cancel/prepare.rs:35-53` 每次 cancel 克隆两个全量 HashSet；
`validation/identity.rs` 每次 commit 重收集全量 id 集。长会话下每次流式冻结都是
O(历史总量)。建议增量维护 id 索引。

#### M-CONV-5. 迟校验：非法内容 freeze 放行、commit 才拒绝，pending turn 只能 cancel ✅ 已修复（M3-6）

`finish_assistant`（`pending/turn.rs:195-239`）不检查同消息内重复 tool_use id（→
`register_tool_calls` 恒失败永久卡死）和 assistant 消息含 Image/ToolResult 块（→
commit 时 `InvalidRoleBlock`，而 `ReadyToCommit` 禁止 ResumeTurn/CommitTurn cancel，
只剩 DiscardTurn，整轮已冻结的 tool 往返作废）。应在 finish_assistant 做与 commit 相同
的块级预检，尽早失败。

#### M-CONV-6. `resolved_provider_call_id` 的唯一性假设比 validation 实际保证的弱 ✅ 已修复（M3-7）

`history/index.rs:413-433`：validation 保证的是"未被 claimed 的 provider id 中唯一"，
构造特定配对可使 index 重推导候选集为 {A, B}——release 下 `expect` 通过但可能取错，
debug 下 `debug_assert!` 直接 panic。restore 接受外部快照的 `None` pairing，路径可达。
应按 validation 相同的 claimed 排除语义重推导。

#### M-CONV-7. fork 静默丢弃父 projection 与 artifacts 📄 已降级（文档承认现状，M3-8）

`boundary/fork.rs:92-95`：child 无条件重建全 raw 投影，父已付费的 summary artifact 不
随 fork 继承（语义正确但成本高），文档未说明此取舍。

### adapter / stream 层

#### M-ADP-1. `StreamEvent::Usage` 语义在两个 adapter 间不一致，契约未文档化

Anthropic 发增量段（`stream/usage.rs:17-35`），OpenAI 终态一次性发完整累计
（`terminal.rs:35`）。Accumulator 靠加法碰巧都正确，但直接消费事件流的上层无法知道
该求和还是取最新。应在文档明确"所有 Usage 事件均为可累加增量"。

#### M-ADP-2. openai_resp 对 `sequence_number` 的强校验对兼容端点过脆

`normalizer/mod.rs:333-345` 要求从 0 严格连续且 wire 结构体必填。真实 OpenAI 兼容
端点（推理网关、代理）常省略或不保证连续，此时整个流以 protocol error 终止。建议缺
字段时跳过校验或仅告警。

#### M-ADP-3. facade 流式 `Done.events` 缺 `ApprovalRequested`，违反跨路径事件契约 ✅ 已修复（M5-2）

非流式用 `weave_approval_events` 织入审批事件（`facade/agent.rs:387`），流式终止输出
直接 `events: collected.events`（`facade/agent/stream.rs:291`）不含审批事件。
`RunOutput.events` 文档承诺与流 yield 一致（`run.rs:144-148`）；现有 parity 测试比较
的是流 yield 序列而非 `Done.events`，漏检。

#### M-ADP-4. `AgentRunStream` 为 `!Send` 且未声明

`stream.rs:87-95`：`Rc<RefCell<&'a mut DefaultAgentMachine>>` + 无 Send bound 的 boxed
future，不能跨 tokio::spawn 移动；`Agent::stream` 文档只字未提，与 `ChatSession::stream`
（Send）语义不一致。

#### M-ADP-5. restore 路径缺与 build 路径同级的声明重名/委托校验 ✅ 已修复（M5-6）

`AgentRestoreBuilder::build` 只做 `ensure_unique_tool_names`（`snapshot.rs:772-776`），
不把 restored delegation 合成的 `ask_<name>` 声明与重新注入的 typed tool 名对撞检查；
`AgentBuilder::build` 有全量校验（`agent.rs:1291`）。restore 后注入同名工具会产生重名
声明发给 provider。

## 🟢 低严重度

### 复制代码（最值得重视的结构性问题）

- **两个 LLM adapter 间大面积逐字重复**：整个 SSE decoder（87/88 行仅注释不同）、
  `validate_event_stream_content_type` + `map_transport_error`、`chat()`/`chat_stream()`
  HTTP 传输样板、header/URL 拼接、`map_transport_error` 共 4 份副本。H-SEC-4 的修复
  要改 4 处。decoder 本就写成完全泛型，提到公共模块零成本。
- **三个 CLI adapter 各复制约 200 行同构代码**：`ProcessTurn`/`ProcessIo`
  （spawn/close/kill/wait）、`drain_and_emit`、`intersect_capabilities` 等。H-EXT-1/
  H-EXT-3/M-EXT-5/M-EXT-6 每个缺陷都要修三遍（实际已在三份里一字不差）。建议抽共享
  的 line-oriented child-process 模块。

### 协议解析边角

- 零参数 function call 的空 `arguments` 被当协议错误（`openai_resp/stream/normalizer/
  item.rs:350` 等），建议空串按 `{}`。
- Anthropic wire 必填字段对兼容端点偏脆（`MessageDelta.usage` 无 serde default 等）。
- 非 JSON 行直接 Protocol 错误杀掉整个 CLI session（decoder.rs:206-207），CLI 升级
  混一行 warning 即全线崩溃，建议容忍 N 个连续非 JSON 行。
- `ContentBlock` 封闭枚举遇未知 block 类型硬失败，与 lib.rs "前向兼容" 承诺冲突，
  缺 `Unknown { raw }` 兜底 variant。
- `Usage` 反序列化对非对象 details 字段硬报错（`usage.rs:158-162`）。

### 并发 / 资源

- Mutex poison 两种策略并存：collab 原语做中毒恢复，trace/budget/facade 十余处
  `.expect("… poisoned")` panic。应统一。
- `NestedMachine` 子节点只增不减（无 detach）；`start_pending_children` 在任何 step
  末尾调用，取消子 A 的同一步会顺带启动无关子 B。
- `CancelRecovery` cursor 永不对外可见但可被持久化快照携带，restore 后
  `begin_user_turn` 的重置不覆盖它 → turn 直接进 Error。
- facade `FacadeApproval.pending` 跨 run 泄漏：stream 被 drop 时审批条目永久滞留。
- budget 预检与 charge 非原子（已事后捕获，可接受但值得文档化）；dispatch 预算耗尽时
  仍降级派 worker，无硬出口。

### 性能

- `TraceHandle::record_node` 每次 O(n) 全表扫重，一个 run 的 trace 是 O(n²)。
- `Plan::add_task` 为环检测整表 clone，每次 O(n)。
- facade 每 run 深拷贝整棵 tool schema 树（`agent.rs:314-319`），可改 `Arc<[Tool]>`。
- `insert_set_against` 为校验整体 clone 两份全量行集，内存峰值 ~3 倍。
- openai 流路径内存放大 2–3 倍（raw.clone + done_item 保留 + unmodeled_events 堆积）。

### API / 文档打磨

- `ask`/`send`/`run` 三动词表同一动作；`Agent` 对标 `ChatSession` 却不带 Session 后缀。
- crate 级文档（lib.rs）完全未提 facade 层，而 README 说它是推荐入口；`facade/mod.rs`
  文档停留在 milestone 叙事（提到不存在的 `AgentSession`）；prelude 漏导出
  `FacadeError` 与全部 model 高频类型。
- `RunEvent` 预置了从不产生的 variant 却承诺 "shape stable"；`#[non_exhaustive]`
  策略不一致（错误类型有、核心枚举没有）。
- `ApprovalRequest::call_id` 用空串当"无 id"哨兵，应为 `Option`。
- `Normalized<T>` 字段全 pub，可构造 value/raw 自相矛盾的值。
- `ToolCall` 无 `extra` 逃生舱，与 `ContentBlock::ToolUse` 不对称。
- 配置校验缺口：空 model 名可通过 build、temperature 接受 NaN/无穷、空 delegate 工具
  名无校验（而 base_url/api_key 反而有 blank 校验，标准不一致）。
- `FacadeIds` 的 "globally unique" 表述过强（跨实例从 1 重新开始）。
- 各种静默覆盖：usage extra 合并、Accumulator metadata 合并。
- worktree 路径可预测（`temp_dir()/agent-lib-worktrees/agent-<id>`）且靠 `path.exists()`
  幂等，共享机器上可被预建目录/符号链接劫持；dirty-close 保留的 worktree 永不清理。
- OpenCode 权限模式塌缩（Prompt/AcceptEdits/Plan 等价于 auto-reject）未显式记录。
- `max_decision_loops` 计数口径（每次 interaction 都计入）与"turn 数"直觉不符，需文档
  明示。

## 修复优先级建议

| 批次 | 内容 | 理由 |
|---|---|---|
| P0 | H-SEC-1~4（安全）、H-EXT-1、H-EXT-3（真机必踩） | 小改动高风险；多处是同一修复在复制代码里做 N 遍，宜先收敛再修 |
| P1 | H-STATE-1~6（状态一致性/正确性 bug） | 破坏数据或静默丢信息，每个都要补回归测试 |
| P2 | M-PROM-*（承诺 vs 实现）、M-ERR-1/2（错误与取消语义） | 决定补实现还是改文档；M-ERR-1 的软拒绝出口是后续健壮性地基 |
| P3 | 复制代码收敛、M-CONV/M-ADP 健壮性、API 打磨 | 0.1.x 窗口期内做，避免固化成兼容负担 |
